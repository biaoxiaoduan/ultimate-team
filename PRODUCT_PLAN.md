# AI Agent 开发团队管理平台产品方案

## 1. 文档信息

- 文档名称：AI Agent 开发团队管理平台产品方案
- 文档版本：v1.1
- 创建日期：2026-03-14
- 更新日期：2026-03-15
- 文档用途：作为后续产品设计、技术设计、开发排期、测试验收的统一基线
- 文档状态：可进入方案评审与架构设计阶段

## 2. 文档目标

本方案基于现有 PRD 进行落地化细化，目标不是继续扩展想法，而是将产品收敛为一个可分阶段交付、可验证价值、可支持后续研发实施的系统方案。

本方案重点解决以下问题：

1. 明确首版产品边界，避免“大而全”导致无法交付。
2. 将模块、流程、数据对象细化到可进入设计和开发阶段。
3. 明确 AI Agent 的职责边界，避免把所有自动化能力都放入首期。
4. 给出推荐技术选型、迭代顺序、验收标准和风险控制建议。

## 3. 产品定位

### 3.1 产品定义

该产品是一个面向 AI-driven 软件研发场景的项目管理与调度平台，用于统一管理项目、需求、任务、文档、研发执行、测试反馈和版本交付过程，并通过 AI Agent 提供任务拆解、调度建议、状态跟踪、报告生成和自动化操作能力。

### 3.2 核心价值

- 将分散在 Git、文档、测试平台、CI/CD 系统中的研发信息集中管理。
- 将“需求 -> 任务 -> 开发 -> 测试 -> 发布 -> 报告”串成闭环。
- 让 AI 从“问答工具”升级为“团队协作中的可执行调度者”。
- 为后续全自动化交付打基础，但首版先保证“人可控、AI 增效”。

### 3.3 目标用户

- 当前阶段：单个独立开发者 / 单人 AI 开发团队
- 未来扩展：小型研发团队、纯 AI Agent 团队、外部客户项目团队

## 4. 产品原则

### 4.1 设计原则

1. 平台先解决协同与可见性，再逐步增强自动化执行。
2. 所有 AI 自动动作必须可追溯、可审计、可回滚。
3. 首版优先支持标准研发流程，不处理高度个性化流程。
4. 先打通最小闭环，再扩展高级功能。

### 4.2 单用户优先原则

首版默认只有一个使用者，因此不将多角色、多人协作、复杂权限控制作为开发前置条件。系统应优先保证单人场景下的高效使用与最小维护成本。

设计要求如下：

1. 所有核心流程都必须支持单人独立完成。
2. 数据模型可以预留 `owner_id`、`created_by` 等字段，但首版不实现复杂权限判断。
3. 页面和接口先按“单用户工作台”设计，避免为了未来多人协作引入不必要复杂度。
4. 角色权限体系作为二期扩展能力预留，不纳入 MVP 验收条件。

### 4.3 首版范围控制原则

首版聚焦以下 4 个问题：

1. 如何统一管理项目、文档、任务和迭代。
2. 如何把 Git / CI / 测试结果关联到任务进度。
3. 如何让 AI Agent 协助拆解任务、生成测试建议和项目报告。
4. 如何让管理者看到项目风险、进度和交付状态。

以下能力不进入首版正式范围：

- 多组织、多租户复杂结算体系
- 完整的低代码流程编排引擎
- 自动写代码并直接合并到主分支的无人审批模式
- 复杂工时核算、绩效考核系统
- 完整的外部应用市场和插件生态

## 5. 产品版本规划

### 5.1 V1（MVP）目标

- 项目创建与基础配置

- 迭代和任务管理

- AI 自动日报 / 周报
- 基础进度看板和项目仪表盘
- 测试用例记录与测试结果回填

### 5.2 V1.5 目标

- GitHub / GitLab Webhook 同步
- Pull Request 关联任务

- 多 Agent 协同调度

## 7. 使用模式与权限策略

### 7.1 首版使用模式

首版产品按单用户模式设计，默认由一个人完成需求管理、任务维护、开发跟踪、测试记录、报告生成和 AI 调度操作。

这意味着首版不处理以下问题：

- 多成员邀请
- 项目内角色分配
- 资源级权限隔离
- 审批流中的角色责任划分

### 7.2 首版权限策略

首版仅保留最小权限能力：

- 支持单账号登录或本地初始化账号
- 默认当前登录用户拥有全部功能权限
- 敏感配置仅对当前账号可见
- 所有关键操作保留审计日志

### 7.3 后续扩展预留

为了后续扩展到团队协作场景，数据结构和接口命名建议预留如下能力：

- User
- ProjectMember
- Role
- Permission
- resource_scope

但这些能力不纳入首版实现范围。

## 8. 核心业务流程

### 8.1 项目启动流程

1. 当前用户创建项目。
2. 填写项目基础信息、技术栈、仓库地址、CI/CD 地址、环境说明。
3. 上传或录入 PRD、需求文档、设计文档。
4. 创建首个迭代并设置目标、周期、里程碑。

### 8.2 需求到任务流程

1. 当前用户创建需求项或导入 PRD。
2. 选择需求范围并触发 AI 拆解任务。
3. AI 输出建议的模块、任务、优先级、依赖关系、验收标准、测试建议。
4. 当前用户审核后生成正式任务。
5. 任务进入待开发池，由当前用户自行认领和推进。

### 8.3 开发执行流程

1. 当前用户开始处理任务。
2. 在任务详情中查看需求、设计、验收标准、关联文档。
3. 提交代码后，系统通过手动录入或 Webhook 关联 commit / branch / PR。
4. CI 运行结果回填到任务和项目面板。
5. 任务开发完成后流转到测试阶段。

### 8.4 测试执行流程

1. 当前用户基于任务创建或补充测试用例。
2. 执行功能测试 / 回归测试。
3. 回填测试结果、缺陷、阻塞原因。
4. 测试通过后任务进入待发布或已完成状态。

### 8.5 报告生成流程

1. 系统每日汇总任务状态、代码提交、测试结果、风险信号。
2. AI 生成日报 / 周报草稿。
3. 当前用户可预览、修订、发送。
4. 报告沉淀到报告中心，可导出和追溯。

## 10. 功能模块详细方案

建立单用户模式下的最小账号能力和系统级基础配置能力。

- 基础登录
- 个人资料与偏好设置
- AI Key / 集成配置管理

#### 二期功能

- OAuth 第三方登录
- 细粒度权限模板
- SSO / 企业账号接入

### 10.2 项目管理

#### 项目字段定义

- 项目名称
- 项目标识 Key
- 项目描述
- 项目状态：规划中 / 进行中 / 已暂停 / 已归档
- 仓库类型：GitHub / GitLab / 其他
- 仓库地址
- 默认分支
- CI/CD 地址
- 前端技术栈
- 后端技术栈
- 数据库
- 中间件
- 部署环境说明
- 项目负责人
- 创建时间 / 更新时间

#### MVP 功能

- 创建、编辑、归档项目
- 配置项目基础信息
- 维护技术架构档案
- 查看项目摘要数据

### 10.3 文档中心

#### 文档类型

- PRD
- 需求说明
- 技术设计
- 测试计划
- 测试用例
- 发布说明
- 会议纪要

#### MVP 功能

- 新建文档元数据
- 富文本或 Markdown 内容编辑
- 文件上传
- 版本记录
- 文档与项目 / 任务关联

#### 规则建议

- PRD 和需求文档必须可关联到迭代或任务。
- 测试文档必须支持与测试用例、测试执行结果关联。

### 10.4 迭代管理

#### 迭代字段

- 迭代名称
- 所属项目
- 迭代目标
- 开始日期
- 结束日期
- 状态：未开始 / 进行中 / 已完成 / 已关闭

#### MVP 功能

- 创建迭代
- 维护迭代目标
- 将任务归属到迭代
- 查看迭代燃尽和完成率

### 10.5 任务管理

#### 任务类型

- Epic
- Feature
- Story
- Task
- Bug
- Test Task

#### 任务核心字段

- 标题
- 编号
- 所属项目 / 迭代
- 来源需求 / PRD
- 任务类型
- 优先级：P0 / P1 / P2 / P3
- 状态：待评审 / 待开发 / 开发中 / 待测试 / 测试中 / 待发布 / 已完成 / 已关闭
- 负责人
- 协作人
- 预计开始 / 截止日期
- 预估工时
- 标签
- 验收标准
- 风险说明
- 关联文档
- 关联分支 / PR / commit / 构建记录

#### MVP 功能

- 手动创建任务
- AI 拆解生成任务建议
- 任务分配与状态流转
- 看板视图
- 列表筛选
- 任务评论和活动记录

- 待评审 -> 待开发
- 待开发 -> 开发中
- 开发中 -> 待测试

- 测试中 -> 待发布
- 待发布 -> 已完成
- 任意状态 -> 已关闭

### 10.6 AI Agent 中心

#### 首版定位

AI Agent 在首版主要做“辅助决策与内容生成”，不直接执行高风险动作。

#### MVP 能力

- 基于 PRD 生成任务拆解建议
- 基于任务和变更内容生成测试点建议
- 生成日报 / 周报 / 风险摘要
- 根据任务状态识别延期风险和阻塞风险

#### 输入来源

- 项目信息
- 文档内容
- 任务数据
- 测试结果
- 集成系统回传状态

#### 输出类型

- 任务列表建议
- 测试建议
- 风险提示
- 报告草稿
- 下一步行动建议

#### 必须具备的控制机制

- 所有 AI 输出默认进入“草稿”状态
- 关键动作必须由人确认
- 保留 prompt、输入摘要、输出结果、确认人、执行时间

### 10.7 测试管理

#### 测试对象

- 测试计划

- 风险任务列表

#### 可视化组件

- 项目进度概览卡片
- 任务状态分布图
- 迭代趋势图
- 风险预警列表

### 10.9 报告中心

## 12. 数据模型建议

建议首版采用 PostgreSQL 作为主数据库，核心实体如下：

### 12.1 核心实体

- User
- Project
- Iteration
- Requirement
- Document
- Task
- TaskComment
- TaskRelation
- TestCase
- TestRun
- Defect
- Integration
- BuildRecord
- Report
- AgentJob
- AgentOutput
- AuditLog

以下实体作为后续多人协作扩展预留，不纳入首版必做范围：

- Role
- ProjectMember
- Permission

### 12.2 关键关系

- 一个 Project 包含多个 Iteration、Document、Task、Integration、Report。
- 一个 Task 可关联一个或多个 Requirement、Document、TestCase、BuildRecord。
- 一个 AgentJob 属于一个项目，也可以关联一个任务、文档或报告。
- 一个 AgentOutput 可转化为 Task、Report、TestCase 草稿。

### 12.3 数据库设计原则

- 业务主表结构稳定，扩展信息放 JSONB。
- 所有外部系统标识保留 `external_id`。
- 关键业务对象保留 `created_by`、`updated_by`、`deleted_at`。
- 状态变更必须有事件日志。

## 13. 非功能需求

### 13.1 安全要求

- 支持账号密码登录，密码加密存储
- API Token / 第三方密钥加密存储
- 关键操作写入审计日志
- 单账号数据默认全量可见
- 限制 AI 读取敏感配置明文

### 13.2 性能要求

- 常规列表页接口响应时间小于 500ms
- 项目仪表盘接口响应时间小于 2s
- AI 报告生成请求在 30s 内返回草稿或排队状态

### 13.3 可用性要求

- 首版目标可用性 99.5%
- 异常任务需支持重试
- AI 调用失败不影响主业务数据操作

### 13.4 可审计要求

- 所有任务状态流转保留日志
- 所有 AI 输出可追溯输入来源
- 所有配置变更保留操作人和时间

## 14. 推荐技术方案

### 14.1 技术选型建议

为了降低研发复杂度并提升交付速度，建议首版统一采用 TypeScript 技术栈：

- 前端：React + TypeScript + Vite
- UI：Ant Design 或基于 Ant Design 的企业后台组件体系
- 状态管理：TanStack Query + Zustand
- 后端：Node.js + NestJS
- 数据库：PostgreSQL
- 缓存 / 队列：Redis + BullMQ
- 文件存储：本地对象存储抽象层，后续可切换 S3 / OSS
- AI 集成：OpenAI SDK
- 鉴权：轻量登录态方案，后续再扩展 RBAC
- 部署：Docker Compose 起步，后续切 Kubernetes

### 14.2 服务划分建议

首版建议采用“模块化单体”而非微服务：

- auth 模块
- user 模块
- project 模块
- document 模块
- iteration / task 模块
- test 模块
- report 模块
- integration 模块
- agent 模块
- audit 模块

这样可以控制系统复杂度，后续如 AI 任务量增长，再拆分 agent-worker。

### 14.3 AI 执行架构建议

- Web API 负责接收请求和展示结果
- 后台队列负责执行 AI 任务
- Agent 结果写入 AgentOutput 表
- 人工审核通过后再同步写入 Task / Report / TestCase

- 状态筛选
- 时间范围筛选
- 关键词搜索
- 按更新时间倒序

## 17. 验收标准

### 17.1 MVP 验收条件

满足以下条件即可认为 MVP 达到可内部试用标准：

1. 可创建项目并完成基础配置。
2. 可录入 PRD / 需求文档并与任务关联。
3. 可创建迭代、任务并在看板中流转状态。
4. 可记录测试用例和测试执行结果。
5. 可生成项目日报或周报。
6. 可通过 AI 输出任务拆解建议和报告草稿。
7. 单用户模式下所有核心功能可独立完成。
8. 所有关键操作可追溯。

### 17.2 核心业务验收场景

- 从项目创建到首个迭代上线全流程可走通
- 一个需求可以拆分为多个任务并在单用户模式下推进到底
- 开发任务可以关联代码与构建记录
- 测试失败能反馈回任务并形成阻塞
- AI 生成内容可被审核后转为正式业务数据

## 19. 风险与对应策略

### 19.1 产品风险

- 风险：首版范围过大导致延期
- 策略：严格冻结 MVP 范围，二期功能单独排期

### 19.2 技术风险

- 风险：外部系统集成差异大
- 策略：首版只做标准化配置和有限集成，不做深度双向同步

### 19.3 AI 风险

- 风险：AI 输出不稳定、存在幻觉
- 策略：首版仅作为辅助草稿，关键动作必须人工确认

### 19.4 组织风险

- 风险：团队对平台使用习惯不统一
- 策略：以项目模板、标准流程、操作日志约束使用方式

## 21. 结论

原始 PRD 的方向是成立的，但如果直接按“大而全的平台”推进，研发很容易失控。更可行的路径是：

- 首版先做“项目管理 + 任务协同 + 文档沉淀 + 测试跟踪 + AI 辅助报告”
- 暂不做高风险的全自动代码执行与发布闭环
- 用模块化单体架构快速打通业务闭环
- 在内部场景验证后，再逐步扩展为更强的 Agent 调度平台

该文档可作为后续产品设计与研发实施的正式输入。
