## 1. 文档信息

- 文档名称：AI Agent 研发编排平台产品方案
- 文档版本：v2.1
- 创建日期：2026-03-14
- 更新日期：2026-03-15
- 文档用途：作为产品设计、系统架构、研发排期、测试验收的统一基线
- 文档状态：待评审

## 2. 产品概述

本产品不是传统意义上的项目管理工具，而是一个面向软件研发交付的 AI Agent 编排平台。

平台的核心目标是把“需求 -> 迭代计划 -> 角色分工 -> 文档产出 -> 开发执行 -> 测试验证 -> 发布准备”这一条研发链路，交给一组可管理、可替换、可协同的 sub-agent 来完成。

每个 sub-agent 对应一个研发角色，例如：

- 产品经理 Agent
- 设计师 Agent
- 开发 Agent
- 测试 Agent
- 发布配置 Agent

每个 Agent 都可以绑定到底层执行提供方，例如：

- Codex
- Claude Code
- 未来扩展的其他代码或任务执行 Agent

平台负责做三件核心事情：

1. 把需求自动拆解成可执行的迭代计划。
2. 把每轮迭代按角色拆解为多个 Agent 任务并驱动协作。
3. 统一管理 Agent 的配置、执行、交接、产物和状态。

## 3. 产品定位

### 3.1 产品定义

该产品是一个单用户优先的 AI 研发编排平台，用于让一个操作者同时管理多个专业角色 sub-agent，并通过统一编排机制驱动这些 Agent 完成软件研发全流程工作。

### 3.2 核心价值

- 将“需求拆解为迭代计划”从人工动作变成平台内核心能力。
- 将“产品、设计、开发、测试、发布”角色抽象为可配置、可替换的 Agent。
- 将 Codex、Claude Code 等执行器统一接入到同一个平台模型中。
- 让 Agent 协作过程可追踪、可审计、可重试、可人工干预。
- 让每轮迭代天然沉淀开发文档、接口文档、测试用例、测试报告和代码提交记录。

### 3.3 目标用户

- 当前阶段：单个操作者管理多个 AI 角色 Agent 的研发团队
- 下一阶段：小型人机协同团队
- 未来阶段：多项目、多团队、多 Agent 编排平台

## 4. 产品原则

### 4.1 设计原则

1. Agent 编排优先于传统项目管理。
2. 需求拆解与迭代规划是核心产品能力，不是附属功能。
3. 人始终保留审批和纠偏权。
4. Agent 必须有明确角色边界和交接产物。
5. 所有执行动作都必须可追踪、可审计、可回放。

### 4.2 单用户优先原则

首版默认只有一个真实操作者，因此首版不做复杂的人类角色权限系统。产品中的“角色”主要指 Agent 角色，而不是人类账号角色。

首版要求：

1. 一个用户可以创建项目、配置 Agent、发起编排、审核输出。
2. 所有 Agent 输出默认可由当前用户查看和确认。
3. 复杂的人类成员协作、组织权限和审批流不纳入 MVP。

### 4.3 编排原则

Agent 协作不能只靠自由聊天或临时 prompt 串联，必须有明确编排机制。

平台采用以下原则：

1. 每次执行必须有一个明确的编排 Run。
2. 每个 Agent 的输入、输出、状态、失败原因都必须记录。
3. Agent 之间通过结构化交接件协作，不直接依赖上下文猜测。
4. 关键阶段必须支持人工确认后再进入下一阶段。

### 4.4 流程优先交互原则

产品首版不能做成“所有模块平铺在一个大页面上”的管理台，而应做成有明确入口、上下文和下一步指引的流程型产品。

交互原则如下：

1. 复杂主任务必须使用独立页面承载，例如需求录入、计划确认、编排执行、交付物审核。
2. 弹窗只用于轻量创建、确认、风险提示和少量字段编辑，不承载长表单和复杂上下文。
3. 抽屉只用于查看关联信息或做局部配置，不替代主流程页面。
4. 每个核心页面都要给出“下一步操作”入口，用户不需要回到首页重新找功能。
5. 关键流程必须支持阶段性返回，但不能让用户在同一页面同时处理所有阶段任务。

## 5. 核心场景

### 5.1 场景一：从 PRD 自动生成迭代计划

1. 用户创建项目并录入 PRD。
2. 平台调用产品经理 Agent 分析需求。
3. Agent 输出迭代拆解建议，包括：
   - 迭代目标
   - 迭代优先级
   - 每轮范围边界
   - 每轮交付物
   - 每轮风险点
4. 用户审核后生成正式迭代计划。

### 5.2 场景二：按迭代启动多角色 Agent 协作

1. 用户选择某轮迭代并发起执行。
2. 编排器依次触发产品经理、设计师、开发、测试、发布配置 Agent。
3. 各 Agent 基于上游交接件生成各自产物。
4. 所有产物沉淀到平台并关联到该轮迭代。

### 5.3 场景三：通过 Codex / Claude Code 执行开发任务

1. 开发 Agent 绑定到底层执行器。
2. 编排器为开发 Agent 下发结构化任务。
3. 执行器返回执行日志、产出摘要、代码变更结果。
4. 平台记录执行过程，并驱动测试 Agent 进入下一阶段。

### 5.4 场景四：Agent 可管理、可替换、可增删

1. 用户可以新增一个 Agent。
2. 用户可以删除或停用一个 Agent。
3. 用户可以修改 Agent 角色、底层执行器、提示词模板、可处理任务类型。
4. 用户可以决定某轮迭代由哪些 Agent 参与执行。

## 6. 核心产品能力

### 6.1 需求拆解与迭代规划

这是 MVP 的核心功能之一。

平台必须支持：

- 输入 PRD、需求文档、补充约束
- 调用产品经理 Agent 输出迭代计划草稿
- 将需求拆解为：
  - 迭代
  - 工作包
  - 角色职责
  - 验收标准
  - 风险项
- 支持人工修改迭代计划并确认落地

### 6.2 多角色 sub-agent 管理

这是 MVP 的核心功能之一。

平台必须支持以下 Agent 角色模板：

- 产品经理 Agent
- 设计师 Agent
- 开发 Agent
- 测试 Agent
- 发布配置 Agent

平台必须支持：

- Agent 列表
- Agent 创建
- Agent 编辑
- Agent 停用 / 删除
- Agent 绑定角色模板
- Agent 绑定执行提供方
- Agent 配置系统提示词 / 任务模板

### 6.3 Agent 执行提供方适配

平台需要抽象 Provider Adapter 层，统一接入不同执行器。

MVP 先支持：

- Codex Adapter
- Claude Code Adapter

每个 Adapter 至少具备：

- 连接配置
- 调用能力
- 输入输出封装
- 执行日志采集
- 失败状态回传

### 6.4 Agent 编排与协同

平台需要一个中央编排器来管理多 Agent 工作流。

它负责：

- 接收迭代执行请求
- 按阶段分配给不同 Agent
- 管理前后置依赖
- 记录 Agent 状态
- 收集交付物
- 失败重试
- 进入人工确认节点

### 6.5 交付物与证据链管理

每轮迭代必须沉淀结构化交付物，包括但不限于：

- 迭代概述
- 开发文档
- 接口文档
- 测试用例
- 测试报告
- 发布配置说明
- 执行日志
- 代码提交记录

这些交付物必须可追溯到：

- 哪个项目
- 哪轮迭代
- 哪个 Agent
- 哪次执行
- 基于哪个输入生成

## 7. MVP 重新定义

### 7.1 MVP 目标

MVP 不是做一个“大而全”的研发平台，而是先验证如下闭环：

1. 用户可以录入需求并生成迭代计划。
2. 用户可以创建、删除、配置多角色 Agent。
3. 平台可以按一轮迭代驱动多个 Agent 顺序协作。
4. 开发 Agent 能通过 Codex 或 Claude Code 执行任务。
5. 测试 Agent 和发布配置 Agent 能产出文档和检查结果。
6. 每轮迭代能沉淀完整文档和执行记录。

### 7.2 MVP 必做模块

1. 基础账号与系统配置
2. 项目与工作区管理
3. 需求中心
4. 迭代规划中心
5. Agent 注册中心
6. Provider 适配与配置
7. 编排中心
8. 交付物中心
9. 基础执行看板

### 7.3 MVP 延后模块

- 多人协作权限系统
- 多组织与多租户
- 自动合并 PR
- 自动生产环境发布
- 自定义复杂工作流拖拽编排
- 完整通知中心

### 7.4 MVP 产品形态

MVP 采用“全局导航 + 项目内流程页 + 轻量弹窗”的产品形态。

核心要求：

1. 用户先进入工作台，而不是直接看到所有模块内容。
2. 进入项目后，以项目级导航在不同阶段页面间切换。
3. 需求、计划、Agent、Run、交付物都以独立页面呈现。
4. 创建类动作可通过弹窗触发，但创建完成后要跳到对应详情页或流程页。
5. 运行中流程必须有明确的当前步骤、上一步产物和下一步入口。

## 8. 角色 Agent 体系设计

### 8.1 产品经理 Agent

职责：

- 分析 PRD
- 输出需求拆解
- 生成迭代计划
- 为其他 Agent 生成目标、范围、验收标准

主要输出：

- 迭代计划
- 工作包列表
- 验收标准
- 风险提示

### 8.2 设计师 Agent

职责：

- 分析需求与迭代目标
- 输出页面结构、交互说明、设计约束
- 生成设计交接件

主要输出：

- 页面清单
- 信息架构
- 交互描述
- 设计说明文档

### 8.3 开发 Agent

职责：

- 基于产品和设计交接件输出开发方案
- 产出开发文档和接口文档
- 调用 Codex 或 Claude Code 执行开发任务

主要输出：

- 开发文档
- 接口文档
- 代码变更结果
- 实现说明

### 8.4 测试 Agent

职责：

- 根据需求、设计、代码变更生成测试用例
- 产出测试计划和测试报告
- 验证核心流程是否通过

主要输出：

- 测试用例文档
- 测试报告

### 8.5 发布配置 Agent

职责：

- 检查配置项和发布前条件
- 输出部署说明、环境变量清单、发布检查项

主要输出：

- 发布配置文档
- 环境清单
- 发布检查报告

## 9. 编排与协作方案

### 9.1 推荐方案

推荐采用“编排引擎 + 事件总线 + 状态存储”的混合方案，而不是只用总线。

原因：

- 只用总线适合解耦，不适合控制复杂阶段状态。
- 只用工作流引擎会让 Agent 间异步事件处理不够灵活。
- 混合方案既能做阶段控制，也能做异步重试和日志追踪。

### 9.2 方案组成

#### 1. Orchestrator

中央编排器，负责：

- 创建 Run
- 生成阶段任务
- 控制依赖顺序
- 处理失败、重试、暂停、继续

#### 2. Event Bus

事件总线，负责在系统内部传播状态变化。

典型事件：

- `requirement.submitted`
- `plan.generated`
- `iteration.started`
- `agent.task.assigned`
- `artifact.generated`
- `approval.requested`
- `agent.task.failed`
- `run.completed`

#### 3. State Store

持久化记录：

- 当前 Run 到哪个阶段
- 哪些 Agent 已完成
- 哪些 Agent 失败
- 哪些交付物已生成
- 哪些审批节点未通过

#### 4. Artifact Store

用于统一保存所有迭代产物、执行日志和结构化输出。

### 9.3 MVP 编排模式

MVP 先采用固定阶段编排：

1. 需求分析
2. 迭代规划
3. 设计交付
4. 开发交付
5. 测试交付
6. 发布准备

每个阶段默认由一个主 Agent 负责，支持人工确认后进入下一阶段。

### 9.4 交接机制

Agent 之间不能只靠自然语言长上下文传递信息，而应通过结构化交接件传递。

交接件最少包含：

- 来源阶段
- 来源 Agent
- 目标 Agent
- 输入摘要
- 关键约束
- 产出链接
- 风险提醒

## 10. 核心业务流程

### 10.1 项目启动流程

1. 用户从工作台点击“新建项目”。
2. 系统打开项目初始化弹窗，录入项目名称、仓库、工作区信息。
3. 创建成功后跳转到项目概览页。
4. 用户在项目概览页看到初始化清单：
   - 配置 Provider
   - 启用 Agent
   - 导入需求
5. 用户按清单进入对应页面完成配置。

### 10.2 需求到迭代计划流程

1. 用户从项目概览页进入需求中心页。
2. 在需求中心点击“新建需求”，弹出录入弹窗。
3. 创建成功后自动跳转到需求详情页。
4. 用户在需求详情页维护版本、目标、约束和验收条件。
5. 用户点击“生成迭代计划”，跳转到计划工坊页。
6. 计划工坊页展示草稿、风险、工作包和角色分工。
7. 用户确认后，系统把计划标记为正式计划，并给出“去配置 Agent”或“发起 Run”的下一步入口。

### 10.3 迭代执行流程

1. 用户从计划详情页或项目概览页进入 Run 中心。
2. 点击“新建 Run”弹窗，选择计划和迭代。
3. 创建成功后跳转到 Run 详情页。
4. Run 详情页按阶段展示：
   - 当前阶段
   - 上游交接件
   - 当前 Agent
   - 可执行动作
5. 用户在当前页完成启动、执行、确认、失败、重试、暂停、继续。
6. 阶段完成后，页面自动聚焦到下一阶段。

### 10.4 交付归档流程

1. Run 完成后，系统从 Run 详情页给出“查看交付物”入口。
2. 用户进入交付物中心的该轮迭代详情页。
3. 系统自动汇总并按分类展示产物：
   - README
   - development.md
   - api.md
   - test-cases.md
   - test-report.md
   - release.md
4. 用户可在该页查看代码提交、测试结果、执行日志和异常信息。
5. 如有缺失项，用户可从该页回到对应阶段进行补充。

## 11. 信息架构

首版应采用两层信息架构，而不是把所有模块堆在一个页面。

### 11.1 全局导航

- 工作台
- 项目
- 执行记录
- 系统设置

### 11.2 项目内导航

进入某个项目后，左侧或顶部二级导航固定为：

- 项目概览
- 需求中心
- 计划工坊
- Agent 中心
- Run 中心
- 交付物中心
- 集成与设置

### 11.3 关键页面职责

- 工作台：展示项目状态、待处理 Run、最近交付、下一步行动
- 项目概览：展示项目基础信息和流程推进清单
- 需求中心：展示需求列表和筛选
- 需求详情页：维护需求内容、版本和计划生成入口
- 计划工坊：查看和确认某条需求对应的迭代计划草稿
- Agent 中心：管理模板、实例、Provider 绑定
- Run 中心：查看所有运行列表
- Run 详情页：执行阶段操作和查看交接件
- 交付物中心：按迭代或 Run 查看最终产物

### 11.4 弹窗与抽屉规则

适合弹窗的动作：

- 新建项目
- 新建需求
- 新建 Agent
- 新建 Run
- 删除确认
- 阶段失败原因填写

适合抽屉的动作：

- 查看 Agent 配置摘要
- 查看交接件摘要
- 查看最近执行日志

不应放在弹窗中的动作：

- 长篇需求编辑
- 迭代计划审核与确认
- Run 阶段执行主视图
- 交付物总览与审核

## 13. 数据模型建议

建议首版使用 PostgreSQL 作为主数据库，核心实体如下：

- User

- IterationPlan

- OrchestrationRun

- ExecutionLog
- BuildRecord
- TestReport
- AuditLog

### 13.1 关键关系

- 一个 Project 包含多个 Requirement、IterationPlan、Iteration、AgentInstance、OrchestrationRun、Artifact。
- 一个 Requirement 可以生成多个 IterationPlan 草稿，但只有一个被确认。
- 一个 Iteration 包含多个 WorkPackage。
- 一个 WorkPackage 可以分配给某个 AgentInstance。
- 一个 OrchestrationRun 包含多个 RunStage。
- 一个 RunStage 可以触发多个 AgentTask。
- 一个 AgentTask 会生成一个或多个 Artifact。
- 一个 Handoff 用于连接两个阶段或两个 AgentTask。

### 13.2 数据库设计原则

- 主实体字段稳定，扩展字段使用 JSONB。
- 所有执行相关对象保留状态机字段。
- 所有重要实体保留 `created_by`、`updated_at`、`version`。
- 所有 Run 和 Agent 执行都必须保留原始输入摘要和输出摘要。

## 15. 推荐技术方案

### 15.1 技术栈

- 前端：React + TypeScript + Vite
- 后端：Node.js + NestJS
- 数据库：PostgreSQL
- 队列 / 异步任务：Redis + BullMQ
- 文档存储：本地文件系统抽象层，后续可切对象存储
- 执行器适配：Provider Adapter Layer
- 前端状态：TanStack Query + Zustand

### 15.2 编排技术建议

MVP 推荐如下实现：

- Orchestrator：NestJS 内部编排服务
- Event Bus：BullMQ 事件或 Redis Streams
- State Store：PostgreSQL
- Artifact Store：数据库元数据 + 本地文件落盘

原因：

- 实现成本低
- 易于本地开发
- 具备基本重试、异步和审计能力

V2 再考虑：

- Temporal
- NATS / Kafka
- 更灵活的可视化工作流引擎

### 15.3 Provider 接入方式

每个 Provider Adapter 负责统一实现以下接口：

- `validateConfig`
- `startTask`
- `pollTask`
- `cancelTask`
- `normalizeOutput`
- `collectLogs`

这样 Agent 不直接依赖具体执行器品牌。

## 17. MVP 验收标准

满足以下条件即可认为 MVP 达到可用标准：

1. 可以录入需求并生成迭代计划草稿。
2. 可以确认正式迭代计划。
3. 可以创建、编辑、删除、停用 Agent。
4. Agent 可以绑定 Codex 或 Claude Code。
5. 可以发起一轮迭代 Run。
6. 平台可以按固定阶段驱动多个 Agent 协同。
7. 可以为每轮迭代沉淀开发文档、接口文档、测试用例、测试报告、发布配置文档。
8. 失败状态可见且可以重试。
9. 所有 Agent 执行记录和交付物可追溯。
10. 用户可以沿页面流程完成主路径，而不是在一个页面处理全部功能。

## 18. 核心验收场景

- 用户提交一个 PRD，平台生成 3 到 6 轮迭代计划草稿。
- 用户确认其中一版计划后，平台生成正式迭代列表。
- 用户为某个项目启用产品经理、设计师、开发、测试、发布配置 5 个 Agent。
- 开发 Agent 可以绑定 Codex，测试 Agent 可以绑定 Claude Code。
- 用户发起某轮迭代，平台依次触发多个 Agent 并沉淀完整交付物。
- 平台能够看到某个阶段失败、重试和最终完成的完整记录。

## 20. 风险与对策

### 20.1 产品风险

- 风险：把平台做成传统项目管理工具，导致核心差异性丢失
- 对策：坚持以“需求拆解 + Agent 编排”作为产品主线

### 20.2 技术风险

- 风险：不同 Provider 行为差异大
- 对策：统一通过 Adapter 层隔离差异

### 20.3 编排风险

- 风险：只靠消息总线导致状态难以控制
- 对策：采用“编排器 + 总线 + 状态存储”的混合方案

### 20.4 交付风险

- 风险：Agent 输出很多，但缺乏结构化交付物
- 对策：强制要求每个阶段产出标准化 Artifact

## 22. 结论

这次产品重设计后的核心变化是：

- “需求拆解为迭代计划”从辅助能力升级为核心能力
- “产品经理、设计师、开发、测试、发布配置”从人类角色转为可配置的 sub-agent
- “Agent 可添加、删除、绑定执行器、参与编排”进入 MVP 主体
- “编排器 + 事件总线 + 状态存储”作为首版推荐架构
- “流程型页面 + 轻量弹窗”成为首版交互基线，不再采用单页功能平铺

该产品的本质不再是项目管理平台，而是一个以软件交付为目标的 AI Agent 编排平台。
