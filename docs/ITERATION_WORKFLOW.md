# 迭代研发工作流

## 1. 文档目的

本文档用于固定 `ultimate-team` 项目的迭代研发方式。后续每次迭代都必须严格按照本文档执行，确保每一轮都具备完整的方案沉淀、接口定义、测试设计、代码实现、测试验证和 Git 提交记录。

本文档是流程基线，不描述某一轮具体需求；具体迭代范围由迭代计划和该轮迭代文档决定。

## 2. 适用范围

适用于本项目 MVP 阶段及之后的所有迭代。

当前默认前提：

- 单用户模式开发与使用
- 单仓库管理
- 本地开发、本地测试通过后再提交 Git
- 每轮迭代都保留完整文档，不覆盖历史版本

## 3. 每轮迭代的固定产物

每次迭代至少要产出以下内容：

1. 迭代概述文档
2. 开发文档
3. 接口文档
4. 测试用例文档
5. 测试结果文档
6. 代码实现
7. 自动化测试代码
8. Git 提交记录

## 4. 文档目录规范

每次迭代使用独立目录保存文档，目录结构固定如下：

```text
docs/
  iterations/
    iteration-01-foundation/
      README.md
      development.md
      api.md
      test-cases.md
      test-report.md
    iteration-02-project-management/
      README.md
      development.md
      api.md
      test-cases.md
      test-report.md
```

目录命名规则：

- 格式：`iteration-XX-<slug>`
- `XX` 为两位数字迭代编号
- `slug` 为英文短名称，使用短横线分隔

## 5. 每份文档的职责

### 5.1 `README.md`

用于说明该次迭代的总体信息：

- 迭代标题
- 迭代目标
- 范围说明
- 不包含内容
- 前置依赖
- 交付结果
- 对应 Git commit message

### 5.2 `development.md`

用于描述该轮功能设计和实现方案，至少包含：

- 背景与目标
- 用户流程
- 页面或模块范围
- 数据模型变更
- 状态流转规则
- 关键交互说明
- 技术实现方案
- 风险点与回滚方案

### 5.3 `api.md`

用于描述该轮新增或变更接口，至少包含：

- 接口列表
- 请求路径
- 请求方法
- 请求参数
- 响应结构
- 错误码和异常场景
- 数据权限或访问约束
- 与前端交互关系

如果该轮无后端接口变更，仍需保留文档，并明确写明“本轮无新增接口，仅复用已有接口”。

### 5.4 `test-cases.md`

用于描述本轮测试设计，至少包含：

- 测试范围
- 不测试范围
- 功能测试用例
- 异常流程用例
- 边界条件用例
- 回归范围
- 自动化测试覆盖目标

### 5.5 `test-report.md`

用于记录本轮测试执行结果，至少包含：

- 测试日期
- 测试环境
- 执行的测试命令
- 通过情况
- 失败情况
- 修复记录
- 最终结论

## 6. 每轮迭代的执行步骤

### 步骤 1：确认迭代范围

开始开发前，先确认本轮迭代目标、范围边界和不做内容。

输出：

- 当前迭代标题
- 当前迭代目标
- 当前迭代交付范围

### 步骤 2：创建迭代文档目录

在 `docs/iterations/` 下为本轮迭代创建独立目录，并初始化以下文件：

- `README.md`
- `development.md`
- `api.md`
- `test-cases.md`
- `test-report.md`

### 步骤 3：先写文档，再写代码

必须按以下顺序进行：

1. 先完成 `development.md`
2. 再完成 `api.md`
3. 再完成 `test-cases.md`
4. 文档确认后再开始功能开发

目的：

- 避免边开发边改需求
- 降低返工
- 保证测试和接口边界清晰

### 步骤 4：实现功能与测试代码

本轮开发时至少同时完成以下工作：

- 业务代码实现
- 单元测试
- 必要的集成测试
- 必要的页面或流程测试

要求：

- 不允许只实现功能不补测试
- 不允许测试代码与迭代范围脱节
- 不允许未完成本轮文档就直接提交代码

### 步骤 5：本地测试与验收

提交前必须完成本地验证。

至少包括：

- 类型检查通过
- Lint 通过
- 单元测试通过
- 相关集成测试通过
- 关键流程人工验证通过

如果当前项目阶段尚未搭建完整工具链，则必须在 `test-report.md` 中明确记录未执行项和原因。

### 步骤 6：补充测试结果文档

开发完成并测试后，更新 `test-report.md`，记录：

- 本轮执行的验证命令
- 实际测试结果
- 已知问题
- 是否达到提交条件

### 步骤 7：提交 Git

只有在以下条件全部满足后，才允许提交：

1. 文档齐全
2. 代码完成
3. 测试通过
4. 测试结果已记录

提交规则：

- commit message 必须使用该轮迭代标题
- 每轮至少一个主提交
- 如果本轮需要多个提交，最终也必须有一个能明确标识该轮交付结果的提交

## 7. Git 工作方式

建议每轮迭代使用独立分支：

- 分支命名：`codex/iteration-XX-<slug>`

推荐流程：

1. 从 `main` 切出当前迭代分支
2. 在迭代分支中完成文档、代码、测试
3. 本地测试通过后提交
4. 自查无问题后合并回 `main`
5. 推送远端

如果当前阶段为了简化流程直接在 `main` 上开发，也必须遵守本文档中的文档和测试要求。

## 8. 每轮迭代的完成定义

一轮迭代完成，必须同时满足：

1. 对应迭代目录已经创建并保留
2. `development.md`、`api.md`、`test-cases.md`、`test-report.md` 已完成
3. 对应功能代码已实现
4. 对应测试代码已实现
5. 本地验证通过
6. 已完成 Git 提交和远端同步

## 9. 不允许的做法

以下行为不允许发生：

- 只写代码，不写文档
- 只写功能，不写测试
- 未本地验证就提交
- 用模糊 commit message 提交
- 覆盖上一轮迭代文档
- 将多个大功能无边界地混入同一轮迭代

## 10. 后续执行约束

从下一轮开始，本项目所有迭代默认遵循本工作流。

如果将来要调整流程，应优先更新本文档，再调整实际执行方式。
