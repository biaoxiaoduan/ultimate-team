# MVP 迭代计划

## 1. 文档目的

本文档用于将新的产品主线拆解为可执行的 MVP 迭代计划。当前产品主线已经调整为：

- 需求拆解为迭代计划
- 多角色 sub-agent 管理
- Codex / Claude Code 执行器接入
- 编排驱动的交付闭环

当前阶段只输出计划，不进入开发实现。待计划确认后，再按 [迭代研发工作流](./ITERATION_WORKFLOW.md) 逐轮落地。

## 2. 迭代拆分原则

本次拆分遵循以下原则：

1. 先搭 Agent 平台底座，再做计划能力，再做编排执行。
2. 需求拆解和 Agent 管理必须进入 MVP 前半段，而不是后置能力。
3. 每轮迭代都必须能形成新的平台能力，而不是只做技术预埋。
4. Agent 执行能力必须晚于计划能力，但不能晚于 MVP 末尾。
5. 交付物沉淀和执行日志是 MVP 必须具备的基础能力。

## 3. MVP 范围假设

当前计划基于以下前提：

- 单用户操作模式
- 多个 Agent 角色协作
- Provider 先接 Codex 和 Claude Code
- 编排模式先采用固定阶段，不做自由拖拽工作流
- Git / 测试 / 发布先做基础关联，不做全自动上线

## 4. 迭代总览

建议将 MVP 拆为 6 轮迭代：

1. 迭代 1：工程底座与 Provider 配置
2. 迭代 2：需求中心与迭代计划生成
3. 迭代 3：Agent 中心与角色模板
4. 迭代 4：编排中心与阶段流转
5. 迭代 5：交付物中心与测试发布闭环
6. 迭代 6：端到端联调与 MVP 收口

这样拆分的原因：

- 迭代 1 先建立所有后续功能的运行基础
- 迭代 2 先把产品最关键的计划能力做出来
- 迭代 3 才让 Agent 真正可管理
- 迭代 4 再做多 Agent 协作
- 迭代 5 补齐可交付性
- 迭代 6 做全链路验证和修复

## 5. 各迭代详细计划

### 5.1 迭代 1：工程底座与 Provider 配置

#### 目标

建立平台工程基础、工作区模型和 Provider 接入底座，为后续 Agent 执行做准备。

#### 包含范围

- Monorepo 或单仓多应用结构
- 前后端初始化
- 数据库连接和迁移机制
- 基础账号与系统设置
- 工作区配置
- ProviderConfig 模型
- Codex / Claude Code 基础配置页
- 健康检查和执行日志基础能力

#### 不包含

- 需求拆解
- Agent CRUD
- 编排 Run

#### 本轮需要产出的文档

- `development.md`：工程结构、Provider 抽象、运行方案
- `api.md`：系统设置、工作区、Provider 配置接口
- `test-cases.md`：初始化、Provider 配置、基础可用性测试

#### 主要研发任务

- 初始化前后端项目
- 配置测试与代码规范工具链
- 落库 User、Workspace、ProviderConfig
- 建立 Provider Adapter 接口
- 实现系统设置和 Provider 配置页面

#### 测试重点

- 项目能本地启动
- 数据迁移可执行
- Provider 配置可保存和读取
- 基础日志链路可用

#### 完成标准

- 平台具备可运行底座
- Codex / Claude Code 配置能力可用

### 5.2 迭代 2：需求中心与迭代计划生成

#### 目标

将“需求拆解为迭代计划”正式做成产品能力。

#### 包含范围

- Requirement 和 RequirementVersion 模型
- 需求录入、编辑、版本管理
- 提交需求给产品经理 Agent
- 生成迭代计划草稿
- 迭代计划编辑与确认
- 每轮工作包基础结构

#### 不包含

- 多 Agent 编排
- Agent CRUD
- 真实开发执行

#### 本轮需要产出的文档

- `development.md`：需求模型、迭代计划模型、生成与确认流程
- `api.md`：需求、版本、迭代计划接口
- `test-cases.md`：需求录入、版本切换、计划生成、计划确认测试

#### 主要研发任务

- 落库 Requirement、RequirementVersion、IterationPlan、Iteration、WorkPackage
- 实现需求中心页面
- 实现迭代计划页面
- 实现计划草稿生成和确认流程

#### 测试重点

- 需求版本切换正确
- 迭代计划草稿可编辑
- 正式计划确认后状态冻结正确

#### 完成标准

- 用户可以从需求得到正式迭代计划

### 5.3 迭代 3：Agent 中心与角色模板

#### 目标

把 Agent 作为一等对象纳入系统，支持创建、删除、绑定执行器和管理角色职责。

#### 包含范围

- AgentTemplate 模型
- AgentInstance 模型
- 内置 5 个角色模板
- Agent 创建、编辑、停用、删除
- 绑定 Provider
- 配置提示词模板
- 配置可处理任务类型

#### 不包含

- 真正的跨阶段自动编排
- 自动代码提交

#### 本轮需要产出的文档

- `development.md`：Agent 模板和实例模型、配置约束
- `api.md`：模板、实例、绑定关系接口
- `test-cases.md`：Agent CRUD、启停、Provider 绑定测试

#### 主要研发任务

- 落库 AgentTemplate、AgentInstance
- 实现 Agent 中心页面
- 内置默认角色模板
- 实现 Agent 和 Provider 的绑定流程

#### 测试重点

- Agent 创建和删除正确
- 停用 Agent 后不可参与后续 Run
- Provider 绑定状态展示正确

#### 完成标准

- 平台可以管理完整的角色 Agent 列表

### 5.4 迭代 4：编排中心与阶段流转

#### 目标

实现多 Agent 之间的顺序协作，让某轮迭代真正可以被多个 Agent 执行。

#### 包含范围

- OrchestrationRun 模型
- RunStage 模型
- AgentTask 模型
- Handoff 模型
- 固定阶段状态机
- 人工确认节点
- 失败、重试、暂停、继续

#### 不包含

- 自定义流程设计器
- 并行复杂拓扑

#### 本轮需要产出的文档

- `development.md`：编排状态机、阶段切换、失败重试机制
- `api.md`：Run、Stage、Task、Handoff 接口
- `test-cases.md`：阶段流转、失败重试、人工确认测试

#### 主要研发任务

- 落库 OrchestrationRun、RunStage、AgentTask、Handoff
- 实现编排运行中心页面
- 实现固定阶段流程
- 实现阶段确认和失败重试逻辑

#### 测试重点

- 阶段顺序正确
- 失败后可重试
- 人工确认未通过时不会进入下一阶段

#### 完成标准

- 平台可以驱动一轮迭代从计划进入多 Agent 协作

### 5.5 迭代 5：交付物中心与测试发布闭环

#### 目标

让每轮迭代真正形成可追溯的交付物和验证结果。

#### 包含范围

- Artifact 模型
- 交付物中心
- 开发文档、接口文档归档
- 测试用例、测试报告归档
- 发布配置文档归档
- Git 提交记录基础关联
- TestReport 和 BuildRecord 基础模型

#### 不包含

- 自动上线
- 复杂 CI/CD 编排

#### 本轮需要产出的文档

- `development.md`：Artifact 分类、归档规则、测试发布交付链路
- `api.md`：Artifacts、TestReport、BuildRecord 接口
- `test-cases.md`：交付物关联、测试报告、发布文档归档测试

#### 主要研发任务

- 落库 Artifact、TestReport、BuildRecord
- 实现交付物中心页面
- 实现交付物和 Run / Agent / Iteration 关联
- 实现测试报告和发布文档展示

#### 测试重点

- 各类交付物归档正确
- 同一轮迭代的产物可完整追溯
- Git 记录和测试报告能正确关联

#### 完成标准

- 每轮迭代具备完整交付包

### 5.6 迭代 6：端到端联调与 MVP 收口

#### 目标

把需求、计划、Agent、编排、交付物完整串起来，验证 MVP 闭环。

#### 包含范围

- 全链路联调
- 失败补偿与异常处理补齐
- 关键页面修复
- 核心指标展示
- MVP 验收场景跑通

#### 不包含

- 多租户
- 高级自定义流程
- 自动生产发布

#### 本轮需要产出的文档

- `development.md`：联调说明、修复项、收口范围
- `api.md`：最终接口清单和变更汇总
- `test-cases.md`：端到端验收用例

#### 主要研发任务

- 跑通主路径
- 修复跨模块问题
- 补足缺失日志
- 补足关键验证

#### 测试重点

- 从需求到正式迭代计划
- 从正式迭代到 Agent 编排执行
- 从执行到交付物归档
- 失败、重试、暂停、继续链路

#### 完成标准

- MVP 验收场景全部通过

## 6. 建议的每轮节奏

建议每轮迭代按以下节奏执行：

1. 第 1 天：补齐本轮文档
2. 第 2-4 天：实现功能和测试代码
3. 第 5 天：联调、补测试报告、提交 Git

如果某轮范围偏大，可延长为 1.5 周，但不要把两轮内容合并。

## 7. 计划确认后的执行方式

待本计划确认后，后续按以下顺序执行：

1. 选择当前迭代
2. 创建该迭代文档目录
3. 先写 `development.md`、`api.md`、`test-cases.md`
4. 再开发功能和测试代码
5. 本地测试通过后补 `test-report.md`
6. 使用该轮迭代标题作为 commit message 提交并推送

## 8. 当前建议

当前最合理的起点是先执行：

- 迭代 1：工程底座与 Provider 配置

原因：

- Provider 和工作区是后续 Agent 执行的基础
- 没有底座就无法验证 Codex / Claude Code 接入模型
- 后续计划、Agent、编排都会依赖这些基础模型

本文档用于评审迭代边界和节奏，不代表已经开始开发。
